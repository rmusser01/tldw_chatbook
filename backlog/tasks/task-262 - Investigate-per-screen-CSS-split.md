## Implementation Notes

Investigation complete — full write-up with all measurements in
`Docs/Design/2026-07-17-css-split-investigation.md` (analyzed against current
dev, where the generated stylesheet is now 15,042 lines / ~2,130 rules).

**Verdict: reasoned NO-SPLIT.** Decisive empirical finding (Textual 8.2.7,
probed with the real stylesheet in the repo venv): `Screen.CSS_PATH` *is*
parsed lazily at first push — but `App._load_screen_css` calls
`Stylesheet.reparse()`, which builds a fresh Stylesheet with an empty parse
cache and re-parses **every loaded source, including the remaining global
bundle, on the UI thread at each screen's first visit** (measured: first push
95.9ms incl. a 66.9ms uncached re-parse of the monolith; second push 23.3ms).
Today's flow never pays this (post-startup widget DEFAULT_CSS additions are
parse-cache hits; real app has only 16-27 sources, measured). A maximal split
(~49% of lines movable) saves ~38ms warm / ~70ms cold at startup but charges
~36-45ms warm back per screen first-visit (~+400-1,000ms aggregate per screen
tour), adds a crash-class UnresolvedVariableError risk ($ds-* variables do not
cross file boundaries — verified), and requires per-screen visual QA.

AC#1 audit: 1,792 class+id tokens across 45 generated modules — 41.5%
single-screen-exclusive, 23.8% shared across screens, 14.8% app-chrome or
screen-unreachable, and 271 tokens (15.1%) referenced nowhere in the repo.
The 4.5k-line `_agentic_terminal` (Console design system) and chat/splash
modules serve the default first-paint screen and are unmovable; the monolith
is generated by `css/build_css.py` (byte-identical rebuild verified), so any
split would be about loading, not authoring.

Recommended instead (recorded in the doc): (1) dead-CSS prune (~1.5-3k lines,
~15-30ms, no mechanics risk — natural continuation of the task-169/253
sweeps); (2) only if the first-paint budget still misses after tasks
285/257/258: a two-phase load — global bundle at boot + ONE deferred
vars-prefixed bundle appended via a single idle reparse after first paint
(same ~70ms cold win, exactly one off-critical-path reparse, needs an
early-navigation force-load guard and a one-time full-app visual QA). AC#3
left unchecked — it applies only if that split-shaped work is ever executed.
