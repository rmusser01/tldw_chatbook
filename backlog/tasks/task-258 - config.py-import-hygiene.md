## Implementation Notes

Approach: reused already-parsed/loaded data instead of re-parsing, in both places the audit flagged, and deferred the one true optional-feeling import (chardet is actually a hard dependency, so this is a pure latency deferral, not a lazy-optional pattern).

1. **Embedded TOML double-parse (config.py ~3837-3844)**: replaced the second `tomllib.loads(CONFIG_TOML_CONTENT)` (`_temp_loaded_config_for_models`) with `copy.deepcopy(DEFAULT_CONFIG_FROM_TOML.get("providers", {}))`. Used deepcopy (not the bare dict reference) so `API_MODELS_BY_PROVIDER`/`LOCAL_PROVIDERS`'s per-provider model lists stay independent objects from `DEFAULT_CONFIG_FROM_TOML`'s tree (matches prior isolation semantics from the independent parse; avoids a hypothetical shared-mutable-list hazard even though no current caller mutates these lists in place).
2. **Double user-config load (config.py `load_settings()` ~658-708)**: removed the two independent `tomllib.load()` calls (the always-empty "primary config" probe at `Config_Files/config.toml` — confirmed dead: not written by any installer/build step, and `pyproject.toml`'s package_data for that directory only globs `*.json`/`*.md` — and the redundant re-read of the same user config file `load_cli_config_and_ensure_existence()` already parsed). `load_settings()` now does `toml_config_data = copy.deepcopy(load_cli_config_and_ensure_existence(force_reload=force_reload)); toml_config_data = decrypt_config_section(toml_config_data)` — value-identical to the old three-way merge (defaults < always-empty primary < user file) since the primary-config merge was always a no-op, and the second `decrypt_config_section` call is a verified no-op on already-decrypted data (kept it anyway, for the case a session password is set after the CLI cache was primed with ciphertext — confirmed by the existing `test_set_encryption_password_invalidates_stale_ciphertext_cache` test, unmodified, still green). All downstream post-processing (STT provider defaults, chat-dictionaries folder creation, chunking-type fallbacks, console coercion) is untouched — only the *source* of `toml_config_data` changed, not its value or the rest of the function body.
3. **chardet (Utils/Utils.py)**: removed the module-scope `import chardet`; added function-local `import chardet` inside `safe_read_file` and `FileProcessor.detect_encoding` (its only two call sites, both otherwise dead code — no callers found anywhere in the repo). chardet is a hard dependency (`pyproject.toml` core deps, not an extra), so no try/except was needed to preserve current absence behavior.
4. **`__init__.py` textual.widgets shim** — SKIPPED. Verified empirically (`python -c "import sys, tldw_chatbook; ..."`) that `'textual' in sys.modules` is `False` at the moment `tldw_chatbook/__init__.py` executes, even on the real app's `import tldw_chatbook.app` path (the parent package's `__init__.py` always runs before the submodule's own top-level imports). Gating the shim on that condition would therefore *always* skip installing the `Static.renderable` compatibility shim for the one case that actually needs it — the running app itself — while doing nothing for the non-UI-consumer case it was meant to help (those consumers still transitively import `tldw_chatbook.config` → DB layer → eventually textual-independent code, so the 53ms is orthogonal). Left unchanged; noted in the report per the task's own "if unclear, SKIP and note" clause.
5. **Tests**: new `Tests/Utils/test_config_import_hygiene.py` (13 tests) — (a) unmocked TLDW_CONFIG_PATH + force_reload integration tests following the T229 `_real_config` recipe, asserting both `get_cli_setting` and `load_settings()`-derived values (including `COMPREHENSIVE_CONFIG_RAW`) see a scratch TOML's contents identically, and that `save_setting_to_cli_config` invalidates both caches without a further force_reload; (b) three parse-count tests — a subprocess-based count of `tomllib.loads()` during a fresh `import tldw_chatbook.config` (must be exactly 1), an in-process monkeypatch proving a forced full reload after import never re-parses the embedded default, and a monkeypatch proving `load_cli_config_and_ensure_existence()` + `load_settings()` called startup-style (default `force_reload=False`, matching config.py's own bottom-of-file sequence) open the user config file exactly once, not twice; (c) two subprocess `sys.modules` assertions (`tldw_chatbook.config` and `tldw_chatbook.Utils.Utils` in isolation, NOT `tldw_chatbook.app` -- see correction below) plus direct functional tests of both chardet consumers and one simulated-absence test pinning the current no-try/except crash-propagation behavior. Verified all 5 behavior-changing tests (the 2 parse-count-at-import/startup tests + the 3 chardet tests) genuinely discriminate: copied the file into a detached `origin/dev` baseline worktree and confirmed they fail there, then pass in this worktree.
   - **Correction during testing**: an early draft of test (b) counted `tomllib.loads` calls during a monkeypatched *force_reload* on an already-imported process and passed on baseline too -- because the embedded-TOML double-parse only ever happened once each, at *import time* (module-level statements), not per force-reload call. Rewrote it as a subprocess test that installs the counting wrapper *before* `import tldw_chatbook.config`. Similarly, an initial "startup" parse-count test used `force_reload=True` for *both* calls and incorrectly asserted 1 read; that pairing legitimately still does 2 reads by design (each `force_reload=True` unconditionally bypasses its own cache) both before and after this change -- the real win only shows up in the *default* `force_reload=False` pairing that config.py's own bottom-of-file startup code actually uses, so the test now matches that exact call shape.
   - **Correction on the chardet subprocess test**: `import tldw_chatbook.app` does *not* keep chardet out of `sys.modules` even after this fix, because `requests/__init__.py` itself does `from chardet import __version__ as chardet_version` at module scope (a dependency-version compatibility check, unrelated to this codebase) and something in the app's import chain imports `requests`. Scoped the assertion to `tldw_chatbook.config` and `tldw_chatbook.Utils.Utils` directly, which is the actual surface this task touches; documented the `requests` finding as a concern in the report.
6. **Measured deltas** (venv Python 3.12.11, isolated scratch HOME, steady-state i.e. config.toml already exists, warm-up run discarded, median of 5 subprocess runs each; detached `origin/dev` @ `a63db6ca` as baseline, removed after measuring):
   - Parse counts (embedded `tomllib.loads` / user-config `tomllib.load`, single `import tldw_chatbook.config`): baseline **2 / 2** -> after **1 / 1**.
   - `import tldw_chatbook.config` wall time: baseline median **145.0ms** (mean 145.4ms) -> after median **116.1ms** (mean 115.5ms) -- **~29ms / ~20% faster**.
   - `import tldw_chatbook.app` wall time: baseline median **672.1ms** -> after median **671.2ms** -- within run-to-run noise, as expected: this task's ~29ms slice is small relative to the app's ~670ms total (dominated by the separately-tracked C1/C2 import-time items: ~469ms tldw_api Pydantic models, ~550ms optional-feature eager imports).
