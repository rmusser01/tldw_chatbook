# Roleplay P1g — Console "What's in play" (dictionaries) design

**Status:** Design approved (brainstorm), pending spec review.
**Program:** Personas→Roleplay redesign. P0 #619, P1a #622, P1b #625, P1c #627, P1d #630, P1e #633, P1f #645 MERGED. This slice = the cross-mode **"What's in play"** summary, deferred from P1e/P1f — surfaced in the **Console** (the live chat), not the Personas authoring workbench.
**Worktree/branch:** `.claude/worktrees/personas-redesign`, `claude/roleplay-p1g-whats-in-play` off dev `3db848e0`.

## Problem

A chat's dictionaries now come from two places — the conversation (P1e, `metadata.active_dictionaries` by id) and the active character (P1f, embedded snapshots in `extensions.chat_dictionaries`) — unified at send time by `Chat_Dictionary_Lib.collect_active_chatdict_entries` (dedup-by-name, conversation-wins, enabled-only). But nothing shows the user **what is actually in play** for the current chat, and there is no place to attach/detach a dictionary from within the chat. P1g surfaces the union in the **Console inspector rail** and lets the user attach/detach *conversation-level* dictionaries there (character dictionaries are read-only — authored in the Roleplay workbench).

## Ground truths (verified at dev `3db848e0` — several corrected a looser first read)

- **The live Console = `ChatScreen` (`UI/Screens/chat_screen.py`) → `ChatWindowEnhanced` → `EnhancedSettingsSidebar`** (the *left settings* sidebar). The Console **inspector rail** is `console-right-rail`, whose content widget is **`ConsoleRunInspector`** (`Widgets/Console/console_run_inspector.py`), a structured rows display with section keys (Provider, Sources, Tools, …). Agent runtime defaults on.
- **The live Console has NO dictionary UI.** `create_chat_right_sidebar` (which holds a dead "Chat Dictionaries" collapsible) is composed **only** by the orphaned `UI/Chat_Window.py` (class `ChatWindow`, instantiated nowhere live). The `chat_right_sidebar_collapsed/_width` reactives + the `#chat-right-sidebar` id ARE live (right-sidebar state), but the dict compose function is not. So P1g's dict surface is **net-new** in the inspector rail; the orphaned dict UIs are left alone (backlog cleanup — see Non-goals).
- **TRAP — the inspector recomposes wholesale.** `ConsoleRunInspector` calls `self.refresh(recompose=True)` (`:131`) and renders from `ConsoleInspectorState.rows`; `ChatScreen` builds that state. The Console ticks frequently and this repo has a documented **sqlite-on-tick** perf problem. Therefore the dictionaries summary **MUST be computed only when the current conversation or active character changes, cached, and rendered from the cache — with zero DB I/O on recompose.** Interactive controls that live inside a recompose widget are fragile, so **attach = a modal picker** (a `ModalScreen`, unaffected by recompose) and **detach = post an intent message → the screen performs the write → updates the cached summary → the inspector recomposes from cache**.
- **The runtime union is the single source of truth.** `collect_active_chatdict_entries(db, conversation_id, char_data) -> List[ChatDictionary]` (`Chat_Dictionary_Lib.py:1146`) flattens the applied dicts' entries; it already encodes dedup-by-name, conversation-wins, enabled-only, never-raises. P1g must not fork this logic (drift = "shown ≠ applied").
- **On-change signal is spread.** `app.current_chat_conversation_id` and `app.current_chat_active_character_data` are set at several `chat_events` sites (conversation load/new/clear, character load/clear). The recompute hook is the load-bearing wiring — preferably a reactive watcher on `current_chat_conversation_id` (verify it is a `reactive`) plus the character-load path, funneled through one `refresh_active_dictionaries_summary()` entry point.
- **Reusable pieces:** P1f's `DictionaryPicker(ModalScreen[int | None])` (`Widgets/Persona_Widgets/dictionary_picker.py`) picks an int dictionary id — reuse it for the Console attach (exclude already-attached conversation dicts). P1e's `LocalChatDictionaryService.attach_to_conversation`/`detach_from_conversation` (+ scope-service wrappers) are the write seam. The **worldbook Console handlers** (`Event_Handlers/Chat_Events/chat_events_worldbooks.py`: `handle_worldbook_add_button`/`remove`/`refresh_active_worldbooks` over the conversation) are the working template for "manage this chat's attachments from the Console."

## Goal / Acceptance

- **AC1 — shared union core (parity by construction).** Refactor `Chat_Dictionary_Lib` so a single core resolves the dict-level union once: `_resolve_active_dictionaries(db, conversation_id, char_data) -> list[dict]`, each row `{"name": str, "source": "conversation"|"character", "enabled": bool, "entries": list[ChatDictionary], "shadowed": bool}` — applying the SAME rules (dedup-by-name, conversation-wins, enabled-only for contribution, never-raises on hostile/malformed content). `shadowed=True` marks a *character* dictionary whose name collides with a conversation dictionary (present but not applied). `collect_active_chatdict_entries` is reimplemented on top of the core (flatten entries of non-shadowed enabled rows) and its behavior is **byte-unchanged** (a regression test pins the pre-refactor output). No change to the send path.
- **AC2 — `summarize_active_dictionaries` (net-new, read-model).** `summarize_active_dictionaries(db, conversation_id, char_data) -> {"dictionaries": [{"name", "source", "enabled", "entry_count", "shadowed"}], "source": "local"}` projects the core's dict-level rows (entry_count = `len(entries)`). Async scope-service wrapper (read action) so the screen calls it uniformly. Real-DB tests assert **parity with `collect_active_chatdict_entries`** (the applied set matches), plus shadowed/enabled/never-raises on malformed embedded content.
- **AC3 — Console inspector "Chat Dictionaries" group (read).** A new section in `ConsoleRunInspector` (registered in its section map) that renders the cached summary for the current chat: one row per dictionary — `name · source (from conversation / from character) · a "shadowed" marker where applicable`; a "No dictionaries in play" empty state; a "no active chat" state when there is no conversation and no active character. Rendered purely from `ConsoleInspectorState` (cached) — **no DB read in the inspector's compose/recompose path** (AC-reported: `console_run_inspector.py` + `ConsoleInspectorState` touch).
- **AC4 — compute-on-change + cache.** The summary is recomputed only when the current conversation or active character changes, via one `ChatScreen` entry point `refresh_active_dictionaries_summary()` (hooked from the conversation-load/new/clear and character-load/clear flows — the reactive watcher on `current_chat_conversation_id` where available), stored on `ConsoleInspectorState`, and fed into the inspector-state build. A test proves the summary updates on conversation switch and does NOT re-query on a plain recompose/tick.
- **AC5 — Console attach/detach (conversation-level).** In the inspector group: an **Attach dictionary…** control opens `DictionaryPicker` (a modal — recompose-safe), filtered to local dictionaries NOT already attached to the current conversation; on pick, `attach_to_conversation(dict_id, current_chat_conversation_id)` runs in a Console `app.run_worker(...)` handler (mirroring the worldbook add handler; `exit_on_error` discipline — no uncaught worker exceptions), then `refresh_active_dictionaries_summary()`. **Detach** on a *conversation-source* row → `detach_from_conversation` → refresh. **Character-source rows are read-only** (no detach — authored on the character). No active conversation → attach is unavailable/no-op with a notify. `ConflictError` → the established "changed since it was loaded" notify + refresh.
- **AC6 — scope.** Personas-owned + Character_Chat dictionary files, PLUS the Console inspector touch (`console_run_inspector.py`, `ConsoleInspectorState`, `chat_screen.py`) and a new Console dictionaries sidebar handler — all AC-reported. No dead-code cleanup (backlog). No character-attach from the Console. No lore/world-books (P2), though the row shape (`{name, source, …}`) is generic enough to host `source: "lore"` later. No change to the send-path applied behavior.

- **Modify** `UI/Screens/chat_screen.py` — `refresh_active_dictionaries_summary()` (compute-on-change → cache on inspector state), hook it from the conversation/character change flows, feed the group into the inspector-state build, and wire the attach picker + detach message handlers.

- **Tests** — `Chat_Dictionary_Lib`: shared-core parity (`collect` unchanged pre/post refactor; `summarize` applied-set == `collect` names; shadowed/enabled/never-raises on malformed embedded content) real-DB; inspector-group render (from state: rows, empty state, no-active-chat state) UI; compute-on-change/cache (updates on conversation switch, no re-query on recompose); Console attach/detach flow (attach via picker over the real seam updates the summary; detach removes a conversation row; character rows have no detach) — real-DB where the seam runs. Parity + compute-on-change-cache + real-seam attach are load-bearing.

## Data flow / error handling

Render: on conversation/character change → `refresh_active_dictionaries_summary()` → `summarize_active_dictionaries(db, current_conv_id, current_char_data)` (one DB read) → cache rows on `ConsoleInspectorState` → inspector recomposes from cache. Recompose/tick re-renders from cache with no DB read. Attach: inspector "Attach…" → `DictionaryPicker` modal → picked dict id → Console worker `attach_to_conversation` → refresh. Detach: row message → screen → `detach_from_conversation` → refresh. Errors: `summarize_active_dictionaries` never raises (skips malformed embedded content, same guards as the runtime union); missing/no conversation → empty/"no active chat" state, attach no-op + notify; `ConflictError` → "changed since it was loaded" notify + refresh; all writes in `app.run_worker` with `exit_on_error` discipline so a bad write never kills the Console.

## Testing

Per the AC list. The `Chat_Dictionary_Lib` tests use a real `CharactersRAGDB` seeded with a real conversation (by-id dicts) + a real character (embedded snapshots), asserting the **summary's applied set equals `collect_active_chatdict_entries`** (the parity that guarantees "shown = applied") and that a name-collision marks the character row `shadowed` while the conversation dict wins. The compute-on-change/cache test guards the recompose-tick perf trap (no DB on recompose). The Console attach/detach test drives the real P1e seam so a shape/flow drift can't false-green.
