# Roleplay P1g — Console "What's in play" (dictionaries) design

**Status:** Implemented (P1g).
**Program:** Personas→Roleplay redesign. P0 #619, P1a #622, P1b #625, P1c #627, P1d #630, P1e #633, P1f #645 MERGED. This slice = the cross-mode **"What's in play"** summary, deferred from P1e/P1f — surfaced in the **Console** (the live chat), not the Personas authoring workbench.
**Worktree/branch:** `.claude/worktrees/personas-redesign`, `claude/roleplay-p1g-whats-in-play` off dev `3db848e0`.

## Problem

A chat's dictionaries now come from two places — the conversation (P1e, `metadata.active_dictionaries` by id) and the active character (P1f, embedded snapshots in `extensions.chat_dictionaries`) — unified at send time by `Chat_Dictionary_Lib.collect_active_chatdict_entries` (dedup-by-name, conversation-wins, enabled-only). But nothing shows the user **what is actually in play** for the current chat, and there is no place to attach/detach a dictionary from within the chat. P1g surfaces the union in the **Console inspector rail** and lets the user attach/detach *conversation-level* dictionaries there (character dictionaries are read-only — authored in the Roleplay workbench).

## Ground truths (verified at dev `3db848e0` — several corrected a looser first read)

- **The live Console = `ChatScreen` (`UI/Screens/chat_screen.py`) → `ChatWindowEnhanced` → `EnhancedSettingsSidebar`** (the *left settings* sidebar). The Console **inspector rail** is `console-right-rail`, whose content widget is **`ConsoleRunInspector`** (`Widgets/Console/console_run_inspector.py`), a structured rows display with section keys (Provider, Sources, Tools, …). Agent runtime defaults on.
- **The live Console has NO dictionary UI.** `create_chat_right_sidebar` (which holds a dead "Chat Dictionaries" collapsible) is composed **only** by the orphaned `UI/Chat_Window.py` (class `ChatWindow`, instantiated nowhere live). The `chat_right_sidebar_collapsed/_width` reactives + the `#chat-right-sidebar` id ARE live (right-sidebar state), but the dict compose function is not. So P1g's dict surface is **net-new** in the inspector rail; the orphaned dict UIs are left alone (backlog cleanup — see Non-goals).
- **The inspector recomposes wholesale, but already models group actions.** `ConsoleRunInspector` calls `self.refresh(recompose=True)` (`:131`) and renders from `ConsoleInspectorState`: **rows** (`state.rows`, grouped by `_ROW_GROUPS` into Static text) AND **actions** (`state.actions`, rendered as **Buttons** per group via `_button_for_action`/`_compose_action`, with `widget_id`s handled by `ChatScreen`). So interactive attach/detach are NOT net-new fragility — they are `ConsoleInspectorAction`s in a "Dictionaries" action group, exactly like the existing Recovery/Next/Approvals actions. Actions are **group-level, not per-row** — so detach is a group action opening a modal, not a per-row button.
- **The Console ticks; the state is rebuilt each time.** `ChatScreen._build_console_inspector_state` (`:4926`, called from 3 sites) builds a **fresh** `ConsoleInspectorState.from_values(...)` on each pass, and the repo has a documented **sqlite-on-tick** perf problem. Therefore the summary **MUST be cached on the SCREEN** (a persistent attr, e.g. `self._active_dictionaries_summary`) and merely **read** into each `_build_console_inspector_state` — **zero DB I/O in the build/recompose path**. It is recomputed only on change (below).
- **On-change is a clean reactive.** `current_chat_conversation_id: reactive[Optional[str]]` (`app.py:2296`) and `current_chat_active_character_data: reactive` (`:2299`) are both **reactives**. The recompute hook is therefore a `watch_current_chat_conversation_id` / `watch_current_chat_active_character_data` (or an equivalent Console-side watcher) funneled through one `refresh_active_dictionaries_summary()` entry point that recomputes the cache and requests an inspector rebuild.
- **The runtime union is the single source of truth.** `collect_active_chatdict_entries(db, conversation_id, char_data) -> List[ChatDictionary]` (`Chat_Dictionary_Lib.py:1146`) flattens the applied dicts' entries; it encodes dedup-by-name, conversation-wins, enabled-only, never-raises — and, subtly, **only claims a conversation dict's name for the dedup set when that dict is enabled** (a disabled conversation dict does NOT shadow a same-named character dict). P1g must not fork this logic (drift = "shown ≠ applied"), and the summary's `shadowed` semantics must match it exactly.
- **Reusable pieces:** P1f's `DictionaryPicker(ModalScreen[int | None])` (`Widgets/Persona_Widgets/dictionary_picker.py`) picks an int dictionary id — reuse it for the Console attach (exclude already-attached conversation dicts). P1e's `LocalChatDictionaryService.attach_to_conversation`/`detach_from_conversation` (+ scope-service wrappers) are the write seam. The **worldbook Console handlers** (`Event_Handlers/Chat_Events/chat_events_worldbooks.py`: `handle_worldbook_add_button`/`remove`/`refresh_active_worldbooks` over the conversation) are the working template for "manage this chat's attachments from the Console."

## Goal / Acceptance

- **AC1 — shared union core (parity by construction).** Refactor `Chat_Dictionary_Lib` so a single core resolves the dict-level union once: `_resolve_active_dictionaries(db, conversation_id, char_data) -> list[dict]`, each row `{"name": str, "source": "conversation"|"character", "enabled": bool, "entries": list[ChatDictionary], "shadowed": bool}`, in **conversation-then-character order** — applying the SAME rules (dedup-by-name, conversation-wins, enabled-only for contribution, never-raises on hostile/malformed content). `shadowed=True` marks a *character* dictionary whose name collides with an **enabled** conversation dictionary (present but not applied); a character dict colliding only with a *disabled* conversation dict is **not** shadowed (it applies) — matching `collect`'s exact name-claiming. `collect_active_chatdict_entries` is reimplemented on top of the core (flatten entries of the non-shadowed enabled rows, in order) and its behavior is **byte-unchanged** (a regression test pins the pre-refactor output, including the disabled-conversation-dict edge). No change to the send path.
- **AC2 — `summarize_active_dictionaries` (net-new, read-model).** `summarize_active_dictionaries(db, conversation_id, char_data) -> {"dictionaries": [{"name", "source", "enabled", "entry_count", "shadowed"}], "source": "local"}` projects the core's dict-level rows (entry_count = `len(entries)`). Async scope-service wrapper (read action) so the screen calls it uniformly. Real-DB tests assert **parity with `collect_active_chatdict_entries`** (the applied set matches), plus shadowed/enabled/never-raises on malformed embedded content.
- **AC3 — Console inspector "Chat Dictionaries" group (read).** A new group in `ConsoleRunInspector` — a `_ROW_GROUPS` heading + rows built from the cached summary: one row per dictionary — `name · source (from conversation / from character) · a "shadowed" marker where applicable`; a "No dictionaries in play" row when the summary is empty; a "no active chat" row when there is no conversation and no active character. Rendered purely from the cached rows on `ConsoleInspectorState` — **no DB read in the inspector's compose/recompose path** (AC-reported: `console_run_inspector.py` + `ConsoleInspectorState` touch).
- **AC4 — compute-on-change + cache (on the screen).** The summary is recomputed only when the current conversation or active character changes — via `ChatScreen`'s reactive watchers `watch_current_chat_conversation_id` / `watch_current_chat_active_character_data` (or the equivalent Console-side change points), funneled through one `refresh_active_dictionaries_summary()` entry point — and cached on a **persistent screen attribute** (e.g. `self._active_dictionaries_summary`). `_build_console_inspector_state` (rebuilt fresh each pass, from 3 call sites) merely **reads** that cached attr to produce the group's rows/actions — **no DB query in the build/recompose path**. A test proves the summary updates on conversation switch and does NOT re-query on a plain recompose/inspector-state rebuild.
- **AC5 — Console attach/detach (conversation-level, via group actions + modal).** The inspector group carries two `ConsoleInspectorAction` buttons (group-level, per the inspector's action model — NOT per-row): **Attach dictionary…** opens `DictionaryPicker` (a `ModalScreen` — recompose-safe) filtered to local dictionaries NOT already attached to the current conversation; on pick, `attach_to_conversation(dict_id, current_chat_conversation_id)` runs in a Console `app.run_worker(...)` handler (mirroring the worldbook add handler; `exit_on_error` discipline — no uncaught worker exceptions), then `refresh_active_dictionaries_summary()`. **Detach dictionary…** opens `DictionaryPicker` filtered to the current conversation's *attached* dictionaries (conversation-source only) → `detach_from_conversation` → refresh. **Character-source dictionaries are read-only** (never offered for detach — authored on the character). No active conversation → the attach action is disabled/no-op with a notify. `ConflictError` → the established "changed since it was loaded" notify + refresh.
- **AC6 — scope.** Personas-owned + Character_Chat dictionary files, PLUS the Console inspector touch (`console_run_inspector.py`, `ConsoleInspectorState`, `chat_screen.py`) and a new Console dictionaries sidebar handler — all AC-reported. No dead-code cleanup (backlog). No character-attach from the Console. No lore/world-books (P2), though the row shape (`{name, source, …}`) is generic enough to host `source: "lore"` later. No change to the send-path applied behavior.

## Data flow / error handling

Render: a conversation/character change fires a reactive watcher → `refresh_active_dictionaries_summary()` → `summarize_active_dictionaries(db, current_conv_id, current_char_data)` (one DB read) → cache rows on the screen attr → request an inspector rebuild → `_build_console_inspector_state` reads the cached rows/actions (no DB) → inspector recomposes from cache. Plain recompose/tick re-renders from cache with no DB read. Attach: the "Attach dictionary…" action button → `DictionaryPicker` modal (attachable dicts) → picked dict id → Console `app.run_worker` handler `attach_to_conversation` → refresh. Detach: the "Detach dictionary…" action button → `DictionaryPicker` modal (the conversation's attached dicts) → picked dict id → `detach_from_conversation` → refresh. Errors: `summarize_active_dictionaries` never raises (skips malformed embedded content, same guards as the runtime union); missing/no conversation → empty/"no active chat" rows, attach disabled + notify; `ConflictError` → "changed since it was loaded" notify + refresh; all writes in `app.run_worker` with `exit_on_error` discipline so a bad write never kills the Console.

## Testing

Per the AC list. The `Chat_Dictionary_Lib` tests use a real `CharactersRAGDB` seeded with a real conversation (by-id dicts) + a real character (embedded snapshots), asserting the **summary's applied set equals `collect_active_chatdict_entries`** (the parity that guarantees "shown = applied") and that a name-collision marks the character row `shadowed` while the conversation dict wins. The compute-on-change/cache test guards the recompose-tick perf trap (no DB on recompose). The Console attach/detach test drives the real P1e seam so a shape/flow drift can't false-green.
