# Non-Obscuring Focus And Selection Design

Date: 2026-05-28
Status: Draft for review
Target branch: `dev`

## Purpose

Standardize focus and selection styling across the Textual app so interactive controls remain readable when focused, selected, active, or clicked.

The current application has an endemic pattern where focus or selection uses heavy outlines, saturated fills, or reverse-style contrast that can cover the label text. The user-facing symptom appears in Library source actions, Console transcript and composer controls, and focused input fields. Settings already demonstrates the desired direction: selected/focused category items keep their label readable and use underline as the primary affordance.

This design creates an app-wide interaction-state contract, an audit workflow, and a staged migration plan. It is not a one-off patch for the attached screenshots.

## Current Code Findings

The relevant style surfaces are TCSS plus inline `DEFAULT_CSS` blocks in Python widgets:

- Global reset: `tldw_chatbook/css/core/_reset.tcss`
- Global button component styles: `tldw_chatbook/css/components/_buttons.tcss`
- Agentic/destination workbench styles: `tldw_chatbook/css/components/_agentic_terminal.tcss`
- Feature overrides: `tldw_chatbook/css/features/*.tcss`
- Top navigation inline widget CSS: `MainNavigationBar.DEFAULT_CSS` in `tldw_chatbook/UI/Navigation/main_navigation.py`
- Shared widget inline CSS: `NavigationButton.DEFAULT_CSS` in `tldw_chatbook/Widgets/base_components.py`
- Media inline widget CSS: `tldw_chatbook/Widgets/Media/media_navigation_panel.py` and `tldw_chatbook/Widgets/Media/media_list_panel.py`
- Dialog inline widget CSS: `tldw_chatbook/Widgets/Evals/sample_browser_dialog.py`
- Built output: `tldw_chatbook/css/tldw_cli_modular.tcss`, regenerated by `python tldw_chatbook/css/build_css.py`

Observed patterns:

- `core/_reset.tcss` applies `outline: heavy $accent` to `*:focus`.
- `components/_buttons.tcss` applies a darker primary background and heavy accent outline to `Button:focus` and `Button:hover:focus`.
- Library source actions use `.library-source-action` as one-row buttons, so the heavy focus frame can visually collide with the label.
- Console transcript action buttons use `.console-transcript-action-button:focus` with `$ds-action-focus` background, which can obscure the control label in cramped rows.
- Console composer input focus can show a long orange frame, making the field feel selected or obstructed rather than merely active.
- Top navigation uses inline Python CSS for `.nav-button`, `.nav-button:focus`, and `.nav-button.is-active`; this surface will not be found by a TCSS-only audit.
- Shared `NavigationButton` uses inline Python CSS for `NavigationButton.active` and `NavigationButton:focus`.
- Media widgets use inline Python CSS for `.media-type-button.active` and `.media-item.selected`.
- Evals sample browsing uses inline Python CSS for `.sample-row.selected`.
- Settings has already introduced the target pattern:
  - `#settings-shell Button:focus, #settings-shell Button:hover:focus` uses readable text, raised background, `bold underline`, and `outline: none`.
  - `Button.settings-category-button.settings-active-section:focus` uses `bold underline`.
  - Tests in `Tests/UI/test_settings_configuration_hub.py` assert that Settings focus styles avoid `reverse` and preserve readable labels.

The existing Settings tests should become the model for a broader contract test suite rather than remaining a Settings-only exception.

## Interaction State Model

The app should distinguish four states that are currently conflated:

1. **Keyboard focus**
   The current interaction target. It is transient and should never imply that a mode or item has been chosen.

2. **Selected or active**
   A persistent location, mode, row, tab, or category. Examples: active Settings category, selected Library mode, active Console transcript row, current top navigation tab.

3. **Pressed or hover**
   A temporary pointer/click affordance. This can be slightly stronger than idle but must remain readable.

4. **Semantic state**
   Disabled, warning, blocked, dirty, running, error, approval required, or unsaved. These states may use semantic color, but focus must not hide the label.

The design rule is simple: focus may emphasize a control, but it must never cover the text, invert contrast unpredictably, or create a frame that dominates the actual content.

## Visual Contract

### Buttons, List Rows, And In-Screen Navigation

Focused buttons and list-like rows should use:

- Existing readable foreground color.
- Panel or raised background, not a saturated action fill.
- `text-style: bold underline` as the primary focus affordance.
- No heavy outline.
- No `reverse` text style.

Selected or active buttons and list-like rows should use:

- `text-style: bold underline`.
- Optional subtle raised background.
- Optional existing textual marker, such as `>`, where the screen already uses that grammar.
- Optional semantic color only when the selected item also represents a warning, dirty, blocked, or error state.

Examples:

- Library source actions: underline the focused action; do not draw a heavy frame over `Open Media`.
- Console transcript action buttons: underline focused `Copy`, `Edit`, `Save`, `Try`, and similar controls.
- Modal action buttons: underline focus, preserve button dimensions.
- Sidebar and category buttons: follow the Settings pattern.

### Inputs, Selects, And Text Areas

Inputs are not buttons. They should use a related but distinct non-obscuring focus treatment:

- Keep a thin border so the field boundary remains obvious.
- Add subtle underline or bottom emphasis on focus.
- Avoid heavy full-field outlines.
- Avoid saturated focus frames that visually read as selection.
- Preserve text, cursor, placeholder, and validation color.

This applies to the Console composer, Settings search, filter fields, form inputs, selects, and text areas.

Invalid or blocked inputs may still use semantic warning/error color, but the focus style must not obscure the typed text.

### Top Navigation

Top-level navigation tabs use a hybrid rule:

- The selected destination tab may keep a subtle fill if the label remains readable.
- Keyboard focus on top navigation should use underline instead of adding a stronger fill or heavy outline.
- Top navigation should not become visually louder than the destination content area.

### Data Tables And Trees

Data-table cursor and tree selection need special handling because they often rely on row highlighting:

- Cursor/focus row: readable text, subtle row background, underline where supported.
- Selected row: readable text, subtle background or marker.
- Avoid reverse text unless a widget cannot support a readable alternative.
- If a native Textual widget cannot express underline cleanly, document the exception and add a regression test for readable text contrast.

## Architecture

Create a small interaction-state foundation rather than screen-specific hacks.

Recommended foundation:

- Add design tokens in `core/_variables.tcss` for focus/selection roles:
  - focus foreground.
  - focus background.
  - focus underline/accent.
  - input focus border.
  - input focus underline.
- Update `core/_reset.tcss` so the global fallback does not impose heavy outlines on every focusable control.
- Update `components/_buttons.tcss` so global `Button:focus` and `Button:hover:focus` follow the underline contract.
- Keep semantic button variants intact, but ensure their focused state remains readable.
- Add focused overrides in `components/_agentic_terminal.tcss` for destination-specific classes that currently bypass the shared button rule.

The foundation must not remove all focus affordances. It should replace the obstructive ones with a quieter, consistent, readable pattern.

Inline `DEFAULT_CSS` blocks must be audited with the TCSS source. The implementation plan must not rely on TCSS-only searches, because app-shell and feature widgets can own focus/active/selected rules in Python files.

Top navigation is part of the PR 1 foundation even though its styles are inline Python CSS rather than TCSS. The implementation plan must inspect and update `MainNavigationBar.DEFAULT_CSS` directly when applying the top-nav hybrid rule.

## Audit Inventory

The first implementation PR should include an audit document listing selectors and their migration status. The initial inventory should cover:

- Global rules:
  - `*:focus`
  - `Button:focus`
  - `Button:hover:focus`
- Shared inline widgets:
  - `NavigationButton:focus`
  - `NavigationButton.active`
- Destination workbench:
  - `.library-source-action`
  - `.library-mode-chip`
  - `.console-transcript-action-button`
  - `.console-transcript-message-selected`
  - Console composer input focus selectors.
  - `#settings-shell Button:focus`
  - `.settings-action-row Button:focus`
- Top navigation:
  - `.nav-button`
  - `.nav-button:focus`
  - `.nav-button.is-active`
  - `.nav-button.is-active:focus`, if added during migration
- Known inline feature widgets:
  - `MediaNavigationPanel .media-type-button.active`
  - `MediaListPanel .media-item.selected`
  - `.sample-row.selected`
- Feature screens with local overrides:
  - Chat.
  - Media.
  - Search/RAG.
  - Embeddings.
  - Tools/Settings.
  - Coding.
  - Dialogs and modals.
  - Sidebars.
  - Trees and data tables.

Each row in the audit should record:

- Selector.
- Owning file.
- Screen or widget.
- Control type.
- Current risk.
- Target state.
- Migration PR or status.
- Verification method.

The audit document should live under `docs/superpowers/qa/non-obscuring-focus-selection/` so the design spec remains stable while the screen-by-screen audit can be updated across multiple PRs.

## PR Strategy

Use multiple PRs.

### PR 1: Foundation And Audit

Scope:

- Add shared non-obscuring focus/selection tokens and base rules.
- Add the audit inventory.
- Apply top-nav hybrid behavior in `MainNavigationBar.DEFAULT_CSS`:
  - selected tab may keep subtle fill.
  - focused tab uses underline without heavy outline or reverse text.
- Migrate the worst shared offenders that are already visible in the screenshots:
  - Global button focus.
  - Library source buttons.
  - Console transcript/action buttons.
  - Input focus treatment for shared inputs, including the Console composer where feasible.
- Preserve Settings behavior as the reference pattern.
- Add TCSS, inline-CSS source contract tests, and focused mounted-widget tests.

Acceptance:

- Shared/global focused buttons and the explicitly migrated PR 1 selectors do not use heavy outlines, saturated action fills, or reverse text as the primary focus style.
- Shared/global focused inputs and the explicitly migrated PR 1 selectors keep a thin border and add subtle underline/bottom emphasis instead of a heavy full-field frame.
- Remaining legacy feature overrides are inventoried and deferred to later screen-group PRs, not silently treated as complete.
- Settings tests still pass and continue to assert readable underline focus.
- Top navigation source tests assert the hybrid rule for `.nav-button:focus` and `.nav-button.is-active`.
- The generated `tldw_cli_modular.tcss` is updated.

### PR 2: Console And Library Screen Review

Scope:

- Review all Console and Library focus/selected/active states screen by screen.
- Verify the attached screenshot cases are fixed.
- Add screenshots for approval.
- Add regression tests for any new screen-specific selectors.

### PR 3: Settings Reference And App-Shell Follow-Up

Scope:

- Confirm Settings remains the reference pattern.
- Review any remaining app-shell or shared inline widget state rules discovered by the PR 1 audit, such as `NavigationButton`.
- Add or update tests for shared widget focus readability that were intentionally deferred from PR 1.

### PR 4+: Legacy Feature Screen Groups

Scope:

- Migrate remaining local overrides in screen groups:
  - Chat and legacy chat widgets.
  - Media and collections/tags.
  - Search/RAG and embeddings.
  - Tools/Settings and coding.
  - Dialogs, modals, trees, data tables, and sidebars.

Each PR should include visual checks for screens with visible changes. Where screenshot harnessing is not practical, use TCSS contract tests plus mounted widget tests.

## Screen Review Workflow

For each screen or screen group:

1. Audit selectors:
   Search for `:focus`, `.active`, `.selected`, `.-selected`, `.is-active`, `--highlight`, `outline`, `reverse`, and strong focus backgrounds.

2. Classify controls:
   Button, list row, nav tab, input/select/textarea, data-table cursor, tree row, modal action, semantic status.

3. Apply the contract:
   - Button/list/nav focus: readable text plus underline, no obscuring fill.
   - Input focus: thin border plus subtle underline.
   - Active/selected: underline plus optional marker or subtle background.
   - Semantic state: preserve meaning, do not obscure label.

4. Verify:
   Use focused tests and screenshots for visible screens. Add TCSS parser tests for broad selectors.

5. Record status:
   Update the audit inventory with migration status and verification evidence.

## Testing Strategy

Add tests at three levels.

### TCSS Contract Tests

Parse TCSS source files and assert:

- Global `Button:focus` does not contain `outline: heavy`.
- Button/list focus rules do not contain `text-style: reverse`.
- Known destination button focus rules contain `underline`.
- Settings reference tests remain intact.
- Input focus rules do not rely on heavy outlines as the only visible affordance.

Existing focus accessibility tests that currently assert `outline: heavy` as the correct behavior must be updated in PR 1. The new contract is not "no visible focus"; it is "visible focus without obscuring labels." Tests should assert visible, readable focus affordances such as underline, thin border, or documented widget-specific fallback.

Because top navigation and several legacy widgets use inline `DEFAULT_CSS`, add Python-source contract tests for the inline surfaces migrated in each PR. PR 1 should cover `.nav-button:focus` and `.nav-button.is-active` in `tldw_chatbook/UI/Navigation/main_navigation.py`. Later screen-group PRs should extend the same pattern to migrated inline widget rules such as `NavigationButton`, Media panels, and Evals dialogs. The top-nav test should assert the hybrid rule: selected top nav may keep subtle fill, but focused nav must include underline and avoid heavy outline or reverse text.

These tests should target source TCSS files, not only the generated CSS.

### Mounted Widget Tests

For key screens:

- Focus Library source actions and confirm labels remain readable.
- Focus Console transcript actions and composer input.
- Focus Settings categories and actions.
- Focus top navigation where test harness support exists.

Mounted tests should check style/class contracts where exact terminal rendering is hard to assert.

### Visual QA

Capture screenshots for:

- Library source browser focused action.
- Console focused transcript/action button.
- Console composer input focus.
- Settings selected category reference.
- Top navigation focus/selected hybrid state.

Screenshots are review evidence, not the only verification.

## Non-Goals

- Do not redesign the full theme system in this work.
- Do not rebuild every widget abstraction before fixing the issue.
- Do not remove focus visibility entirely.
- Do not make all selected states identical if semantic state needs to remain visible.
- Do not couple the fix to Console only.
- Do not change behavior or routing of screens as part of the foundation PR.

## Open Risks

- Some Textual widgets may not support the exact underline/focus combination cleanly. Those cases need documented exceptions and readable fallback tests.
- Legacy feature CSS may override shared rules with high specificity. The audit inventory must track these explicitly.
- Generated CSS can drift if source TCSS is edited without running `build_css.py`; implementation plans must include regeneration and diff checks.
- Over-correcting global focus could harm accessibility. The replacement must remain visible enough for keyboard users.

## Success Criteria

- The attached Library, Console, and input focus examples no longer obscure text.
- Settings remains visually consistent and continues to demonstrate the reference underline pattern.
- Buttons, list rows, and in-screen navigation share a readable focus/selection language.
- Inputs use a distinct non-obscuring focus treatment.
- Top navigation uses hybrid selected/focused behavior.
- The audit inventory makes remaining screen-by-screen migrations explicit.
- Future regressions are caught by TCSS and mounted widget tests.
