## Ground truths (verified in code this session — several are traps)

- **`export_json` (`local_chat_dictionary_service.py:540-554`)** returns `{"dictionary_id", "data": {name, description, content, entries, max_tokens, enabled, version}, "source"}` with entries via `ChatDictionary.to_dict()` — **P1c fields (enabled/case_sensitive/priority) already round-trip**, in **to_dict naming** (`key`/`content`, probability int 0-100). **TRAP: `strategy` is NOT in the export**, and `import_json` never passes strategy to save — **JSON round-trips currently lose the strategy** even though `save_chat_dictionary` accepts a `strategy` kwarg (verified signature). This slice fixes both seam sides.
- **`import_json` (`:521-538`)**: `name = data.get("name") or payload.get("name") or "Imported Dictionary"` — **`data.name` WINS**, and our own exports always carry it → the conflict-rename retry MUST mutate `data["name"]`, not pass a payload-level name (which would be silently ignored). Entries go through `_entry_from_payload` (the hardened coercion boundary). Duplicate names → `save_chat_dictionary` raises `ConflictError` (UNIQUE name), propagated uncaught — the UI owns the rename/cancel flow.
- **`import_markdown` (`:491-506`)**: `payload["name"]` is **REQUIRED** (KeyError if absent — no default) → the markdown import path always supplies a name (filename stem; conflict-retry mutates this payload name). Entries via `_parse_markdown_entries` (key/content only; `ChatDictionary` constructor defaults + coercion cover the rest). Grammar: `key: value` lines, `key: |` multi-line blocks terminated by `---@@@---` — exactly mirrors `export_markdown`'s writer.
- **`export_markdown` (`:508-519`)**: returns the stored `content` verbatim when present, else reconstructs `key: content` lines. **Fields LOST: everything except key+content** — type/regex, probability, group, max_replacements, timed_effects, AND the P1c trio. The lossy warning names this list.
- **Versions backend is fully real** (`:556-619`): `list_versions` summaries (`{revision, action, name, created_at}`, no snapshot), `get_version` (full snapshot incl. entries + settings), `revert_version` (optimistic-locked `update_chat_dictionary` + a `"revert"` history record). History = JSON side-file (`app.py:3036`, `get_user_data_dir()/tldw_chatbook_chat_dictionary_history.json`), atomic tmp-then-replace writes, **unbounded growth (documented known issue, not fixed here)**. A lazy `"baseline"` record is created on first history read → **the Versions tab always shows ≥1 row**; label the action honestly.
- **`get_statistics` (`:621-628`)** is thin: `{dictionary_id, entry_count, enabled, source}` — the Stats tab enriches client-side from the already-loaded entries (zero backend changes).
- **Import dialog reuse target verified**: `EnhancedFileOpen` file picker (`Widgets/enhanced_file_picker.py`) + the screen's `_io_dialog_active` refuse-reentry worker idiom (`_open_import_dialog`/`_import_dialog_worker`, `personas_screen.py`) — the dictionaries import clones this pattern with `.json`/`.md` filters.
- The rail's Import button is mode-gated (`set_mode`: characters-only) with a character-specific tooltip; the screen's import action branch returns early for non-characters. Both get dictionaries branches.
- `get_user_data_dir` lives in `Utils/paths.py:101` (UI-appropriate import; `config.py` has a duplicate — don't use it from widgets).

## Goal / Acceptance

- **AC1 — JSON export/import round-trips EVERYTHING.** `export_json`'s data block gains `"strategy"`; `import_json` passes `strategy=data.get("strategy") or "sorted_evenly"` into `save_chat_dictionary`. A round-trip test exports a dictionary with non-default strategy + entries exercising all ten entry fields, imports it (renamed), and asserts field-by-field equality (dictionary-level: name aside, description/content/strategy/max_tokens/enabled; entry-level: all ten in API naming via `get_dictionary`).
- **AC2 — Import flow.** Rail Import un-gates for dictionaries mode (mode-aware tooltip); the screen's import action gets a dictionaries branch cloning the `EnhancedFileOpen` worker idiom (filters: JSON `.json`, Markdown `.md`/`.markdown`). Dispatch by extension: `.json` → read file → `json.loads` (parse failure → notify, nothing created) → `import_json({"data": parsed["data"] if "data" in parsed else parsed})` (accepts both our export envelope and a bare data object); `.md` → `import_markdown({"name": <filename stem>, "content": <file text>})`. On `ConflictError`: a rename-or-cancel dialog pre-filled with the P1a-pattern disambiguation (`"Name (imported)"`, `"… (imported 2)"` probed against the loaded list); retry mutates `data["name"]` (JSON) / `payload["name"]` (markdown). On success: re-render rows and select the imported dictionary (which auto-runs the validation panel over it).
- **AC3 — Export flow.** Settings tab gains **Export JSON** and **Export Markdown** buttons. Both write to `get_user_data_dir()/exports/` (mkdir parents) as `{slug(name)}-{YYYYMMDD-HHMMSS}.{json,md}`; the written path is surfaced in the detail status line (copyable). Markdown export first shows a `ConfirmationDialog` naming the dropped fields (regex/type, probability, group, max replacements, timed effects, enabled, case-sensitivity, priority); JSON exports without ceremony. Copy near the buttons notes exports read **saved** state. Write failures → notify, no partial file left behind (write to a temp path in the same directory, then atomic rename — the history side-file's established pattern).
- **AC4 — Versions tab.** New TabPane listing `list_versions` summaries (revision · action · name · created_at; the ever-present lazy `"baseline"` row labeled as such). **View** loads `get_version` and shows a snapshot summary (entry count + description/strategy/max_tokens/enabled at that revision) in the tab's detail area — no entry table (snapshot entries are in to_dict naming; a count is honest and cheap). **Revert…** uses the screen's existing `ConfirmationDialog` worker idiom → `revert_version` → full reload (rows, detail, validation recompute); concurrent-edit `ConflictError` → the established "changed since it was loaded" copy. Tab content refreshes on selection and after revert/import.
- **AC5 — Stats tab.** New TabPane combining `get_statistics` (entry_count, enabled) with client-side enrichment from the loaded entries: per-type counts (literal/regex), disabled count, approximate token total (`len(content.split())` sum — the engine's own formula), priority min/max when any non-zero. Recomputed with `update_entries` (same hook as validation).
- **AC6 — carried riders.** (a) `validate_entries` gains `malformed_probability` (field `probability`): emitted when the value is present but non-coercible — replacing the current silent `None` swallow; the display layer's `_prob_pct` fallback stays. (b) `PersonasDictionaryDetailWidget.clear()` also clears the validation panel's options and stored findings (symmetry fix).
- **AC7 — scope guard.** Personas-owned + Character_Chat dictionary files only (the seam changes in AC1 are Character_Chat files — in scope). No shared-shell files; no `tldw_api` changes expected (report if one proves trivially required). No attachments work (P1e).

## Data flow / error handling

File reads/writes and service calls run in the screen's existing worker patterns (`personas-io` group, `_io_dialog_active` refuse-reentry). JSON parse errors, path-validation failures, unreadable files → notify + no side effects. `ConflictError` on import → rename dialog loop (cancel = nothing created). Export write failure → notify; tmp-then-rename prevents partial files. Revert conflicts → standard copy. The widget never performs I/O — all service/file work stays on the screen.

## Testing

The fake's shapes must mirror the verified real ones exactly — including the `data.name`-wins precedence (a fake that reads the payload name first would green-light the rename-retry bug the real service would reject).
