## Problem

A dictionary cannot be attached to a conversation through any working UI. The chat send path *reads* a conversation's `metadata` JSON key `active_dictionaries` to decide which dictionaries apply — but **nothing can write that key**, three dead code paths pretend to, and the reverse ("which conversations use this dictionary") is a stub returning `None`. P1e makes attach/detach real on the verified runtime contract, surfaces the reverse used-by, and removes the misleading dead paths.

## Ground truths (re-verified at HEAD `b7bc52e4` — several are traps)

- **Runtime contract:** `chat_events.py:985-996` reads `conv_details['metadata']` JSON → `active_dictionaries` (a **flat list of int dictionary ids**, iterated in stored order via `cdl.load_chat_dictionary(db, dict_id)`; a missing/deleted id is tolerated — skipped). **No per-attachment priority** is read. P1e writes ONLY this key.
- **TRAP — metadata is NOT writable today.** `CharactersRAGDB.update_conversation(conversation_id: str, update_data, expected_version)` (`ChaChaNotes_DB.py:4951`) whitelists **only `title` and `rating`**; a `metadata` field is silently ignored. So the read contract has **no functioning writer anywhere** — the orphaned CCP handler's `update_conversation(..., {'metadata': ...})` write (`conv_char_events.py:3554`) was doubly-dead (uncomposed AND a no-op). P1e's first task makes `metadata` an updatable field (inheriting the method's version-bump / last_modified / sync_log / optimistic-lock machinery).
- **Typing seam:** conversation ids are **strings** (`update_conversation(conversation_id: str, ...)`, `get_conversation_by_id(conversation_id: str)`, the Personas controller's `_open_conversation_id: str`); dictionary ids are **ints**. `active_dictionaries` holds int dict ids. Attach adds `int(dictionary_id)`; conversation ids are never int-cast.
- **TRAP — the existing `ConversationSelectionDialog` is NOT reusable.** `Widgets/conversation_selection_dialog.py` is a **TTS/audio-generation** dialog (confirm = `generate_audio()`, TTS-shaped result) and it does `int(radio_id.replace("conv-radio-", ""))` — casting conv ids to int, which breaks on real string ids. P1e builds a small dedicated attach picker instead.
- **No service attach surface exists.** `chat_dictionary_scope_service` / `local_chat_dictionary_service` have zero attach/detach/link/used-by methods; `list_dictionaries(include_usage=True)` is a stub (`local_chat_dictionary_service.py:305`) returning `{"conversation_count": None}`. P1e builds the seam.
- **The local service's DB is `CharactersRAGDB`** (`self.db`, `app.py:3036`) — it has `get_conversation_by_id`, `update_conversation` (optimistic-locked), `get_connection()` for raw reads, and `search_conversations_page`. The service can implement attach/detach/used-by against it.
- **Three dead attach paths** (approved for removal): the orphaned CCP handlers (`conv_char_events.py` `handle_ccp_dict_apply_button_pressed`/`..._remove_from_conv_...`/`populate_active_dictionaries_list` — registered but their widgets `ccp-dict-apply-button`/`ccp-active-dictionaries-list` are never composed); the non-functional `Event_Handlers/Chat_Events/chat_events_dictionaries.py` module (constructs `ChatDictionary(app.db)` → TypeError; calls 6 nonexistent methods; widgets uncomposed; `refresh_active_dictionaries` invoked live on conversation lifecycle as a silent no-op); and the two dead junction module functions `associate_dictionary_with_conversation`/`get_conversation_dictionaries` (`Chat_Dictionary_Lib.py:1285,1321`, zero live callers). The junction TABLE DDL stays (dropping it is a migration — out of scope).

## Goal / Acceptance

- **AC1 — conversation metadata is writable.** `CharactersRAGDB.update_conversation` gains `metadata` in its updatable-fields whitelist: a provided `metadata` value (expected: a JSON string) is written, version bumps, `last_modified`/sync_log update as for `title`. A real-DB test round-trips a metadata blob through `update_conversation` and re-reads it via `get_conversation_by_id`, asserting persistence + version increment. No change to the `title`/`rating` paths.
- **AC2 — attach/detach service seam (net-new).** On `LocalChatDictionaryService` (wrapped by the scope service with `mode="local"`), read-modify-write the conversation's `active_dictionaries` via `update_conversation`:
  - `attach_to_conversation(dictionary_id: int, conversation_id: str) -> dict`: load conv → parse metadata (missing/malformed → `{}`) → append `int(dictionary_id)` to `active_dictionaries` if absent (dedup, preserve order) → write with the read version. **Idempotent** (already-attached → no-op success). Missing/deleted conversation → `ValueError`. Stale version → `ConflictError` propagates.
  - `detach_from_conversation(dictionary_id: int, conversation_id: str) -> dict`: remove the id if present → write. Not-attached → no-op success.
  - Returns `{"dictionary_id", "conversation_id", "active_dictionaries": [...], "source": "local"}`.
- **AC3 — reverse used-by (replaces the stub).** `list_dictionary_conversations(dictionary_id: int) -> {"conversations": [{"conversation_id": str, "title": str}], "source": "local"}`. Mechanism: a **read-only** query `SELECT id, title FROM conversations WHERE deleted = 0 AND metadata LIKE '%active_dictionaries%'` (parameterized `LIKE` prefilter to shrink the scan), then JSON-parse each candidate's metadata and include it only when `int(dictionary_id)` is an **exact member** of the parsed `active_dictionaries` list — never a substring test (the id-1-matches-11 trap). Malformed metadata on a row → skip that row.
- **AC4 — Attachments tab.** The Detail `TabbedContent` gains an **Attachments** pane (`Entries · Settings · Stats · Versions · Attachments`): a list of the conversations this dictionary is attached to (title + a **Detach** button per row, keyed by conversation id), an empty state ("Not attached to any conversation yet."), and an **Attach to conversation…** button opening the dedicated picker (AC5). The tab recomputes on dictionary select and after any attach/detach. The widget is I/O-free — it emits intent messages (`DictionaryAttachRequested`, `DictionaryDetachRequested(conversation_id: str)`) and renders what the screen feeds via `load_attachments(rows: list[dict])`; the screen owns all service calls.
- **AC5 — dedicated conversation picker.** A new small modal (`Widgets/Persona_Widgets/dictionary_attach_picker.py`) listing conversations (title + a search `Input`, backed by the screen fetching `search_conversations_page`/`search_conversations_by_title`), single-select, returning the selected **string** conversation id via `dismiss(conversation_id)` (or `None` on cancel). No int-casting of ids. NOT the TTS dialog.
- **AC6 — remove the dead attach code.** Delete: the three orphaned CCP handlers + their `CCP_BUTTON_HANDLERS` entries; the `chat_events_dictionaries.py` module + its `CHAT_DICTIONARY_BUTTON_HANDLERS` wiring (`chat_events.py:4923`), the `refresh_active_dictionaries` call sites (`chat_events.py:2309/2385/3204`), the dead `event_dispatcher.py:72` reference, and the dead CSS (`_sidebars.tcss:404`, `tldw_cli_modular.tcss:643`); the two junction module functions. A removal-safety check confirms nothing live imports/dispatches the removed symbols before deleting. Keep the junction table DDL.
- **AC7 — scope.** Personas-owned + Character_Chat dictionary files, PLUS: the `update_conversation` whitelist change + the used-by read query (conversation-metadata touch — **AC-clause reported**), and the dead-code removal in `Chat_Events`/`conv_char_events`/`event_dispatcher`/CSS/DB-module-functions. No character-level attachment, no "What's in play" (P1f); the inspector's Console-attach block reason for dictionaries stays as-is (staging a dictionary to Console is a distinct capability, not conversation-attach, and remains out of scope). No table drop.

## Data flow / error handling

Attach: Detail posts `DictionaryAttachRequested` → screen opens the picker (worker, `_io_dialog_active`) → on a picked conversation id, `scope.attach_to_conversation(dict_id, conv_id)` → refresh the Attachments tab (via `list_dictionary_conversations`). Detach mirrors it from the row button. All service/DB work is on the screen in `personas-io` workers; the widget never does I/O. `ConflictError` (a conversation edited concurrently) → the established "changed since it was loaded" copy; missing conversation / malformed metadata → notify + no-op; the tab tolerates an empty/failed fetch by rendering the empty state.

## Testing

Per the AC list. The service tests use a real `CharactersRAGDB` seeded with real conversations (not just the dictionary fixtures) so attach actually round-trips through the patched `update_conversation` and the used-by scan hits real metadata — this is the level where the "metadata silently ignored" trap and the "id-1-vs-11" trap both live, so a fake alone would give a false green (the lesson from every prior phase: fakes mirror shapes, real-DB tests catch contract bugs).

## Scope / non-goals

- **P1f (next):** character-level attachment (net-new data model — a character-card metadata key or a new junction — PLUS a chat_events read-side merge so character-attached dicts apply to all its chats) + cross-mode "What's in play" (a read summary spanning conversation + character attachments). **Backlog:** per-attachment priority/ordering (the junction has an unused `priority` column; the runtime doesn't read it); dropping the dead junction table (migration); a real `include_usage` count on the list view.
- No change to the runtime read path (`chat_events.py:985-996` stays); P1e only makes the existing contract writable + surfaces it.
