# Roleplay P1f — character-level dictionary attachment (design)

**Status:** Implemented (P1f).
**Program:** Personas→Roleplay redesign. P0 #619, P1a #622, P1b #625, P1c #627, P1d #630, P1e #633 MERGED (Dictionaries mode + conversation-attach complete; dev at schema **v20**). This slice = **character-level dictionary attachment (core)**; the cross-mode "What's in play" summary is decomposed to **P1g** (a read-only view spanning conversation + character attachments — its own cycle).
**Worktree/branch:** `.claude/worktrees/personas-redesign`, `claude/roleplay-p1f-character-attach` off dev `2e696aff`.

## Problem

A dictionary can be attached to a *conversation* (P1e) but not to a *character*. A character-level attachment makes a character's dictionaries apply to **all** of that character's chats (its "always-on" vocabulary) and — the user's explicit intent — travels with the character card when exported/imported, the way an embedded lorebook (`character_book`) does under CCv3. No mechanism links dictionaries to characters today: the runtime send path reads only *conversation* metadata, and character cards embed lorebook content but nothing dictionary-related.

## Ground truths (verified at dev `2e696aff` — several shape the design)

- **Runtime contract (conversation-only today):** `chat_events.py:985-996` reads `conv_details['metadata']` JSON `active_dictionaries` (flat list of int dict ids), loading each via `cdl.load_chat_dictionary(db, dict_id)` and `extend`-ing its entries into `chatdict_entries`. This is the **single** dictionary-load site (the regeneration/other call sites pass `chatdict_entries=None` — regeneration does not apply dictionaries today; pre-existing, out of scope). P1f adds a **second source** here (AC-reported touch).
- **The active character is already in memory at send time:** `app.current_chat_active_character_data` carries the card including `extensions` and its DB `id` (`chat_events.py:2491/2746`); the existing `character_book` handling already reads `extensions` (`:642-651`). So character-embedded dicts need **no DB query** at send — they are read from the loaded card.
- **Serialization inverse pair:** `export_json` (`local_chat_dictionary_service.py:559`) serializes entries via `ChatDictionary.to_dict()`; `load_chat_dictionary` parses via `ChatDictionary.from_dict()` (`Chat_Dictionary_Lib.py:1046`). They are exact inverses on the same stored entry shape. So an embedded `export_json` data block re-parses into the **same `ChatDictionary` objects** the conversation path already produces — the runtime "adapter" is just `from_dict`, not bespoke serialization.
- **`extensions` is already writable + locked:** `update_character_card(character_id, card_data, expected_version)` (`ChaChaNotes_DB.py:3853`) whitelists `extensions` (a JSON field) and is optimistic-locked (version bumps on success). **No schema migration is needed** (contrast P1e's v18→v19; a junction table would have forced v20→v21).
- **Names are the portable key:** `chat_dictionaries.name` is `UNIQUE NOT NULL`, so a dictionary is locally identifiable by name — the key for embedded snapshots and dedup.
- **Export already carries `extensions`:** `character_card_formats.py:192` includes `'extensions': char_data.get('extensions', {})` in the exported card `data`. Unlike `character_book` (promoted to top-level `data.character_book`), a non-CCv3-standard key correctly stays inside `extensions`. So `extensions['chat_dictionaries']` travels tldw→tldw with **no export-path change**.
- **TRAP — `extensions` is co-owned by the character editor.** `PersonasCharacterEditorWidget` batch-saves by starting from a *base copy* of the loaded record and preserving keys it does not edit (e.g. `character_book`). An instant attach that writes `extensions['chat_dictionaries']` directly makes that base stale → a later editor Save would (safely, via the optimistic lock) `ConflictError`, or clobber the attach if it wrote back the stale base. P1f keeps the editor's base coherent (AC5).
- **TRAP — embedded content is untrusted on import.** An imported card's `extensions['chat_dictionaries']` is hostile input (P1d hit three crash bugs on the untrusted-file surface). The runtime must parse it defensively (AC2).
- **Attach seam to mirror:** P1e's `attach_to_conversation` / `detach_from_conversation` / `list_dictionary_conversations` on `LocalChatDictionaryService` (+ async `ChatDictionaryScopeService` wrappers). P1f adds the character-side siblings on the same service.
- **UI home:** the characters mode mounts its detail widgets in `#personas-detail-stack` (`personas_screen.py:451-455`: `PersonasCharacterCardWidget`, `PersonasCharacterEditorWidget`, …). The new dictionaries panel mounts here. The characters detail is a card + batch-Save editor form (no tabbed detail like the dictionaries mode).

## Design decisions (from brainstorm)

- **Storage = embedded content snapshot.** A character owns snapshots of dictionaries in `extensions['chat_dictionaries']`: a JSON **list**, each element an `export_json`-shaped data block (`{name, description, content, strategy, max_tokens, enabled, entries[...]}`, entries in `to_dict` naming). Portable — travels with the card. **Snapshot semantics:** editing the source dictionary later does **not** update the character's copy (identical to `character_book`).
- **Identity = dictionary name** (UNIQUE). Dedup and detach key on name; a character cannot embed two dicts of the same name.
- **Interaction = instant, view-side** (mirrors P1e), not batched into the editor Save.
- **Runtime = additive union.** Character-embedded dicts apply to all the character's chats; conversation dicts add to them. Dedup by name; **on a name collision the conversation's (live, by-id) dict wins** over the character's snapshot.

## Goal / Acceptance

- **AC1 — character attach/detach service seam (net-new).** On `LocalChatDictionaryService` (+ async `ChatDictionaryScopeService` `mode="local"` wrappers):
  - `attach_to_character(dictionary_id: int, character_id: int) -> dict`: take `export_json(dictionary_id)["data"]` (the portable content block — name/description/content/strategy/max_tokens/enabled/entries) → load character (`get_character_card_by_id`) → parse `extensions` (missing/malformed → `{}`) → append that block to `extensions['chat_dictionaries']` **iff no element already has that name** (dedup, idempotent) → `update_character_card` with the read version. Missing character → `ValueError`; stale version → `ConflictError` propagates. Returns `{"dictionary_id", "character_id", "dictionary_name", "character_dictionaries": [names...], "source": "local"}`.
  - `detach_from_character(character_id: int, dictionary_name: str) -> dict`: remove the element whose `name` matches → write. Not present → no-op success.
  - `list_character_dictionaries(character_id: int) -> {"dictionaries": [{"name", "entry_count", "enabled"}...], "source": "local"}`: read-only summary derived from the embedded content only (snapshots are self-contained; no source-dict lookup). Malformed elements are skipped.
  Real-DB tests round-trip through `update_character_card` (embed + dedup + version bump), including missing-character and stale-version.
- **AC2 — runtime union (AC-reported `chat_events.py` touch).** At the send path (`:~984-998`), dedup is at the **dictionary** level (by name), so it must run *before* entries are flattened into `chatdict_entries` (the current loop flattens per-dict entries and discards which dict each came from — that grouping must be preserved to dedup). The loop is restructured to: (a) load each conversation dict (`load_chat_dictionary` → `{name, enabled, entries}`), recording its **name**; (b) load the active character's embedded dicts via a shared helper `Chat_Dictionary_Lib.load_character_dictionaries(char_data) -> list[dict]` — each `{"name": str, "enabled": bool, "entries": list[ChatDictionary]}`, parsing `active_char_data['extensions']['chat_dictionaries']` via `from_dict`, **defensively** (each malformed element skipped, never raising into the send path); (c) **skip any character dict whose name already appeared among the conversation dicts** (conversation-wins), then `extend` `chatdict_entries` with the entries of every surviving **enabled** dict. **Conversation-only behavior is unchanged** — a chat with no active character (or a character with no embedded dicts) produces the exact same `chatdict_entries` as today. A real-send test proves a character-embedded dict actually fires, that a same-named conversation dict wins the collision, that a disabled embedded dict is skipped, and that malformed embedded content degrades to skip (no crash); a regression test pins conversation-only output unchanged. No change to *how* entries are processed once collected (the P1b/P1c engine is untouched).
- **AC3 — instant attach UI (character view).** A new I/O-free `PersonasCharacterDictionariesWidget` mounted in `#personas-detail-stack`: lists the selected character's embedded dictionaries (name + entry-count + a **Detach** button per row, labeled as an **embedded copy** — a snapshot, not a live link), an empty state ("No dictionaries attached to this character yet."), and an **Attach dictionary…** button opening the picker (AC4). It renders what the screen feeds via `load_character_dictionaries(rows: list[dict])` and emits intents `CharacterDictionaryAttachRequested` / `CharacterDictionaryDetachRequested(dictionary_name: str)`; the widget performs no I/O.
- **AC4 — dictionary picker (net-new).** A new small modal `Widgets/Persona_Widgets/dictionary_picker.py` `DictionaryPicker(ModalScreen[int | None])` listing local dictionaries (name + a search `Input`), single-select, returning the chosen **int** dictionary id via `dismiss(dictionary_id)` (or `None` on cancel). The list **excludes dictionaries already attached to this character** (by name) so a pick is never a silent no-op (attach is idempotent-by-name); refreshing a stale snapshot is done by detach + re-attach (which re-snapshots current content — the manual re-sync path). Distinct from P1e's `DictionaryAttachPicker` (which picks a conversation and returns a string id).
- **AC5 — screen wiring + editor coherence.** `personas_screen.py`: `@on(CharacterDictionaryAttachRequested)` opens the picker (worker, `_io_dialog_active` refuse-reentry) → `attach_to_character` → refresh; `@on(CharacterDictionaryDetachRequested)` → `detach_from_character` → refresh; the panel refreshes on character select and after attach/detach. **After any attach/detach, patch the character editor's in-memory base** so a subsequent editor Save preserves the attachment without a version conflict **and** without discarding the user's in-progress form edits: the screen re-reads the character (`get_character_card_by_id`) and calls `PersonasCharacterEditorWidget.sync_attached_dictionaries(chat_dictionaries: list[dict], new_version: int)`, where `chat_dictionaries` is the **full embedded content list** (not the name-summaries the panel renders) — the editor sets `_character_data['extensions']['chat_dictionaries']` and `_character_data['version']` from it, touching nothing the form has edited. `ConflictError` (concurrent edit) → the established "changed since it was loaded" copy. All service/DB work runs in `personas-io` workers; the widget never does I/O.
- **AC6 — portability.** No export/import code change is required for `chat_dictionaries` to travel (it rides inside `extensions`). The **runtime** parses embedded dicts defensively (AC2), so an imported card with malformed/foreign `chat_dictionaries` never crashes a send. No re-sync/refresh-from-source action in P1f (a P1g/backlog nicety); snapshots are frozen at attach time. A test round-trips a character with embedded `chat_dictionaries` through export→import and asserts the snapshots survive.
- **AC7 — scope.** Personas-owned + Character_Chat dictionary/character files, PLUS the runtime union in `chat_events.py` (AC-reported). No conversation-attach change (P1e stays). No "What's in play" cross-mode summary (P1g). **No schema migration; no new character↔dictionary table.** The duplication / `sync_log` cost of embedding content (the character sync triggers log full `extensions`) is accepted as inherent to the portable-snapshot model and documented.

## Data flow / error handling

Attach: the panel posts `CharacterDictionaryAttachRequested` → the screen opens the picker (worker, `_io_dialog_active`) → on a picked dict id, `scope.attach_to_character(dict_id, char_id)` (export_json → embed with name-dedup → `update_character_card`) → patch the editor base (`sync_attached_dictionaries`) → refresh the panel via `list_character_dictionaries`. Detach mirrors from the row button. Runtime: the send path collects conversation dicts (by id) and character dicts (from the loaded card's `extensions`), dedups whole dicts by name (conversation-wins), then flattens the surviving **enabled** dicts' entries into `chatdict_entries` and processes as today. Errors: missing character → notify + no-op; malformed embedded content → skipped defensively (never crashes a send); `ConflictError` → "changed since it was loaded"; the panel tolerates an empty/failed fetch with its empty state.

## Testing

Per the AC list. The service + runtime tests use a real `CharactersRAGDB` seeded with real characters + dictionaries so attach round-trips through the patched `extensions` and the embedded snapshot actually re-parses and **fires in a send** — this is where the `to_dict`/`from_dict` shape assumption and the "does it actually apply" contract live, so a fake alone would give false green (the recurring lesson). The editor-coherence test guards the co-owned-`extensions` trap; the malformed-embedded test guards the untrusted-import surface.

## Scope / non-goals

- **P1g (next):** cross-mode "What's in play" — a read-only summary spanning conversation + character (+ eventually lore) attachments for the current chat. **Backlog:** re-sync/refresh a character's embedded dict from its (edited) source; per-attachment enable/disable at the character level; a size guard/warning when embedding very large dictionaries.
- No change to conversation-attach (P1e). No new table or migration. No change to *how* dictionary entries are processed once collected (P1b/P1c engine unchanged). No cross-tool portability guarantee (a tldw-specific `extensions` key that other card tools ignore).
